# XH-202609 通用网络安全智能体可执行任务清单

> 目标：把“审阅与落地计划”拆成可直接执行的工单，按优先级推进，确保项目从“可演示”稳定过渡到“可交付”。

## 使用方式

- `P0`：必须先做，直接影响稳定交付和回归验证。
- `P1`：交付感增强项，决定作品完整度。
- `P2`：能力增强项，适合在主链路稳定后继续推进。

### P0 任务拆解

#### P0-01 统一启动方式、端口与环境变量

- 统一后端默认端口为 `8000`。
- 统一前端代理回 `http://127.0.0.1:8000`。
- 统一开发环境变量说明。
- 确保脚本、文档、前端配置三者一致。

#### P0-02 前端冒烟测试

- 登录成功后跳转 Dashboard。
- 打开多智能体页，确认页面可渲染。
- 打开告警页，确认列表能拉取。
- 打开用户页，确认管理员可见数据。

#### P0-03 关键 API 回归测试

- `/health`、`/health/detailed`
- `/api/v1/auth/login`
- `/api/v1/dashboard`
- `/api/v1/users`
- `/api/v1/alerts`
- `/api/v1/multi-agent/status`
- `/api/v1/multi-agent/tasks`

#### P0-04 文档收口

- 删除已弃用的本地大模型部署叙述。
- 统一“Phase 4”与当前版本表述。
- 明确哪些是已完成，哪些是规划中。

### P1 任务拆解

#### P1-01 / P1-02 报告导出

- 复用现有报告内容生成逻辑。
- 增加 HTML 到 PDF 的导出能力。
- 增加 Markdown 到 DOCX 的导出能力。
- 前端提供下载按钮和导出状态反馈。

#### P1-03 多智能体实时日志

- 任务创建请求显示在首屏日志中。
- 每个步骤回传显示 Agent、耗时、结果摘要。
- 错误步骤显示可读错误信息。

#### P1-04 / P1-05 业务联动

- 告警详情加载时读取相关资产信息。
- 多智能体研判结果可以回写到告警研判字段。
- 告警列表、资产详情、多智能体结果形成闭环。

## 建议排期

### 第 1 周

- 完成 `P0-01` 到 `P0-05`
- 形成第一版稳定回归基线
- 确保项目能“一次启动、一次登录、一次演示”

### 第 2 周

- 完成 `P1-01` 到 `P1-06`
- 让作品具备更完整的交付感
- 让演示、导出、联动三个点真正闭环

### 第 3 到第 4 周

- 推进 `P2-01` 到 `P2-06`
- 补齐数据联动、外部情报、监控和存储能力
- 为后续扩展留出稳定接口

## 交付验收口径

完成全部 `P0` 任务后，项目应满足：

- 可按统一脚本启动。
- 登录、Dashboard、多智能体、告警、用户管理等关键页面都能打开。
- 关键 API 有自动化回归。
- 文档和代码状态一致。

完成 `P1` 任务后，项目应满足：

- 报告可导出。
- 多智能体执行过程清晰可见。
- 告警、资产、多智能体之间存在可追踪联动。

完成 `P2` 任务后，项目应满足：

- 数据、情报、监控、存储能力基本完整。
- 项目可从“功能成型”升级为“工程化成熟”。

## 风险提示

- 不建议在当前阶段重新引入本地大模型部署线，容易把资源重新拉回硬件瓶颈。
- 不建议在没有 P0 回归测试的情况下继续扩展新功能，否则回归成本会快速升高。
- 不建议同时推进过多 P1/P2 项目，最好先把 `P0` 稳住，再做功能增强。
